AI SQL Agent - Complete Project Documentation
=============================================

1. Project Overview
-------------------

This project is a full-stack "Ask your database" application.

Frontend:
- React
- TypeScript
- Vite
- Tailwind CSS
- shadcn/Radix UI components

Backend:
- Python
- FastAPI
- Google Gemini API
- Neon PostgreSQL
- psycopg PostgreSQL driver

Main purpose:
The user asks a natural-language question like "show all customers from India".
The frontend sends that question to the backend.
The backend asks Gemini to understand the question, inspect the PostgreSQL schema, generate a safe SQL query, execute it on Neon PostgreSQL, and return the SQL plus table results.

The project now uses Neon PostgreSQL, not SQLite.

6. Backend API Routes
---------------------

Defined in backend/apis.py.

GET /health
- Returns backend status.
- Shows whether the agent is initialized.
- Shows whether the Neon/PostgreSQL database is connected.
- Useful first check while debugging.

Example response:

{
  "status": "healthy",
  "agent_initialized": true,
  "database_available": true,
  "database_provider": "neon"
}

GET /databases
- Returns the configured Neon database information.
- The app now uses one configured DATABASE_URL instead of multiple uploaded databases.

Example response:

{
  "databases": ["neondb"],
  "current": "neondb",
  "provider": "neon",
  "connected": true
}

POST /ask
- Main natural-language endpoint.
- Frontend sends the user's natural language question here.
- Backend passes the question to SQLAgent.
- SQLAgent may call schema tools, generate SQL, execute SQL, and return results.

Request body:

{
  "question": "show all customers",
  "show_reasoning": true
}

POST /execute-sql
- Used by the SQL Editor tab in the frontend.
- Takes raw SQL from the user.
- Validates it first.
- Allows only read-only SELECT/WITH queries.
- Executes it on Neon PostgreSQL.

Request body:

{
  "question": "SELECT * FROM customers LIMIT 10"
}

GET /database/view
- Used by the "View Database" button in the frontend header.
- Returns all user tables, columns, constraints, row counts, and sample rows.
- Reads PostgreSQL metadata through information_schema.

POST /upload-data
- Used by the "Upload Data" button in the frontend header.
- Accepts CSV files only.
- Rejects files larger than 5 MB.
- Imports the CSV as a new PostgreSQL table in Neon.
- Uses the CSV filename as the base table name.
- Sanitizes table and column names into PostgreSQL-safe identifiers.
- Stores uploaded columns as TEXT.
- If a table name already exists, creates a unique suffixed table name.

POST /upload-database
- Kept only for compatibility.
- Returns an error because SQLite file upload is no longer supported.

POST /databases/{db_name}/select
- Kept only for compatibility.
- Returns an error because runtime database switching is disabled.
- To change database, update DATABASE_URL in backend/.env.


7. Backend Configuration
------------------------

backend/config.py

Important values:

GEMINI_API_KEY
- Loaded from environment.
- Required for natural-language queries.
- If missing, SQL editor and database viewer can still work, but /ask will return an error.

DATABASE_URL
- Loaded from DATABASE_URL or NEON_DATABASE_URL.
- Required for Neon PostgreSQL connection.

DATABASE_NAME
- Display fallback name.
- Defaults to "Neon PostgreSQL".

MODEL_NAME
- Current value: gemini-2.5-flash-lite
- Gemini model used by SQLAgent.

TEMPERATURE
- Current value: 0.1
- Low value makes generated SQL more consistent.

MAX_QUERY_TIMEOUT
- Query timeout in seconds.

DEFAULT_LIMIT
- Default LIMIT added to queries that do not already have one, unless the query is an aggregation like COUNT/SUM/AVG.

MAX_UPLOAD_SIZE_BYTES
- Current value: 5 * 1024 * 1024.
- Maximum CSV upload size.
- Larger files return "Data size is larger than 5 MB".

BLOCKED_KEYWORDS
- Prevents destructive or unsafe SQL operations.
- Includes INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, TRUNCATE, GRANT, REVOKE, VACUUM, CALL, DO, COPY.


8. SQLAgent
-----------

File:

backend/src/agent.py

Main class:

SQLAgent

Main method:

process_question(question, show_reasoning=True)

What SQLAgent does:

1. Reads Gemini settings from config.py.
2. Creates DatabaseTools.
3. Checks whether Neon/PostgreSQL is connected.
4. Builds a system prompt for Gemini.
5. Defines function-calling tools for Gemini:
   - get_schema_info
   - explore_data
   - get_table_stats
   - execute_sql
6. Receives a natural-language question.
7. Decides whether the question looks like a SQL/data request.
8. Sends the question to Gemini.
9. If Gemini calls a tool, SQLAgent executes that tool locally.
10. Sends the tool result back to Gemini.
11. Repeats until Gemini returns a final text response.
12. Returns a normalized response to the frontend.

The agent supports two modes:

Database connected:
- Gemini can inspect schema.
- Gemini can generate PostgreSQL.
- Backend validates and executes the SQL.
- User receives real data.

Database not connected:
- Gemini can still generate SQL based on assumptions.
- Queries cannot be executed.
- Frontend shows databaseAvailable=false.

Gemini key missing:
- SQLAgent still initializes database tools.
- /ask returns a clear error saying GEMINI_API_KEY is missing.
- SQL editor and database viewer can still be used if DATABASE_URL works.


9. Gemini Tool Calling
----------------------

SQLAgent exposes these tools to Gemini.

get_schema_info(table_name=None)
- If table_name is empty, returns all user tables.
- If table_name is provided, returns columns, types, primary keys, defaults, and foreign keys.

explore_data(table_name, column_name=None, sample_size=5)
- If column_name is empty, returns sample rows.
- If column_name is provided, returns distinct count, non-null count, total count, and sample values.

get_table_stats()
- Returns row count for every user table.

execute_sql(sql)
- Validates the SQL first.
- Runs the SQL in a read-only PostgreSQL transaction.
- Returns columns, rows, row_count, and execution_time.


10. DatabaseTools
-----------------

File:

backend/src/tools.py

Main class:

DatabaseTools

This is the database access layer.

It now uses:

psycopg

not sqlite3.

Important methods:

_connect()
- Connects to Neon PostgreSQL using DATABASE_URL.
- Sets statement_timeout safely using set_config.
- Stores current database name.
- Sets available=True if connection succeeds.
- Stores connection_error if connection fails.

get_schema_info(table_name=None)
- Reads metadata from information_schema.tables, information_schema.columns, and constraint metadata.
- Excludes pg_catalog and information_schema system schemas.

explore_data(table_name, column_name=None, sample_size=5)
- Safely quotes identifiers.
- Returns sample rows or column stats.

get_table_stats()
- Counts rows for every user table.

validate_sql(sql)
- Blocks multiple statements.
- Blocks dangerous keywords.
- Allows only SELECT and WITH.
- Adds LIMIT 100 to non-aggregate queries without LIMIT.

execute_sql(sql)
- Starts a read-only transaction.
- Executes SQL.
- Converts rows to JSON-friendly dictionaries.
- Converts Decimal/date/datetime/time values into frontend-safe values.

get_database_view()
- Builds the structure shown by the frontend "View Database" dialog.
- Returns database name, dialect, tables, columns, row counts, and sample rows.

upload_csv_data(filename, content)
- Creates a new PostgreSQL table from uploaded CSV data.
- Requires a header row.
- Converts the filename into a safe table name.
- Converts headers into safe unique column names.
- Stores every uploaded column as TEXT.
- Inserts all CSV rows into the new table.

close()
- Closes the PostgreSQL connection.


11. SQL Safety Rules
--------------------

The app is designed for read-only querying.

Allowed:
- SELECT ...
- WITH ... SELECT ...

Blocked:
- INSERT
- UPDATE
- DELETE
- DROP
- ALTER
- CREATE
- TRUNCATE
- GRANT
- REVOKE
- VACUUM
- CALL
- DO
- COPY

Also blocked:
- Multiple SQL statements in one input.

Important:
Raw SQL from the SQL editor now goes through validate_sql before execution.

16. Frontend Natural Language Flow
----------------------------------

Files involved:

frontend/src/pages/Index.tsx
frontend/src/components/QueryInput.tsx
frontend/src/hooks/useQuerySimulation.ts
frontend/src/services/api.ts
frontend/src/components/StatusBanner.tsx
frontend/src/components/ReasoningPanel.tsx
frontend/src/components/ResultTable.tsx
frontend/src/components/SQLViewer.tsx

Flow:

1. User types a question in QueryInput.
2. QueryInput calls onSubmit.
3. Index passes runQuery from useQuerySimulation as onSubmit.
4. useQuerySimulation sets status to "thinking".
5. useQuerySimulation calls askQuestion from services/api.ts.
6. askQuestion sends POST /ask to FastAPI.
7. FastAPI calls SQLAgent.process_question.
8. Backend returns success/error, SQL, summary, rows, columns.
9. useQuerySimulation transforms backend response into QueryResponse.
10. Index renders StatusBanner and ResultTable.

24. Current Data Model Assumption
---------------------------------

The app does not hard-code a specific table model.

Instead:
- It connects to whatever Neon PostgreSQL database DATABASE_URL points to.
- It asks information_schema for tables and columns.
- Gemini uses that schema to decide what SQL to generate.

If /database/view returns no tables:
- The database connection works.
- But the connected database has no user tables in visible schemas.
- Or your tables are not in a schema currently included by the app.

Current schema scan excludes:
- pg_catalog
- information_schema

It includes:
- public
- other non-system schemas

Table names outside public are returned as:

schema_name.table_name


25. Common Problems and Fixes
-----------------------------

Problem:
Only two APIs show in FastAPI docs.

Cause:
You ran:

uvicorn server:app --reload

Fix:
Run:

uvicorn apis:app --reload --port 8000


Problem:
GET / returns 404.

Cause:
apis.py does not define a root / route.

Fix:
Use:

http://localhost:8000/docs
http://localhost:8000/health


Problem:
Database not connected. Running in fallback SQL-generation mode.

Possible causes:
- DATABASE_URL missing from backend/.env.
- You are running Uvicorn from the wrong folder.
- psycopg is not installed.
- Neon URL is invalid.
- Network cannot reach Neon.
- Database server is paused/unavailable.

Useful checks:

cd backend
python test_setup.py

or:

.\venv\Scripts\python.exe -c "from dotenv import load_dotenv; load_dotenv(override=True); from src.tools import DatabaseTools; db=DatabaseTools(); print(db.available, db.database_name, db.connection_error)"


Problem:
Database connects but /database/view shows zero tables.

Cause:
The connected Neon database currently has no user tables, or your tables are in a schema not expected by your prompt/data.

Fix:
Create/import tables in Neon, then click Refresh/View Database again.


Problem:
Upload Data says "Data size is larger than 5 MB".

Cause:
The selected CSV file is above the allowed size.

Fix:
Use a smaller CSV file, split the data, or increase MAX_UPLOAD_SIZE_BYTES in backend/config.py and frontend/src/services/api.ts.


Problem:
Upload Data says "Only CSV files are supported".

Cause:
The current upload feature imports CSV files only.

Fix:
Export your data as .csv before uploading.


Problem:
Frontend cannot call backend.

Check frontend/.env:

VITE_API_URL=http://localhost:8000

Restart frontend after changing .env.


Problem:
Natural-language query fails but SQL editor works.

Cause:
DATABASE_URL may be fine, but GEMINI_API_KEY is missing or invalid.

Fix:
Add valid GEMINI_API_KEY to backend/.env and restart backend.


Problem:
Build/test fails with esbuild spawn EPERM.

Cause:
Windows/security/permission issue launching esbuild from node_modules.

What was observed:
- TypeScript check passed.
- ESLint ran.
- Vite build/test failed because esbuild process spawn was blocked.

Possible fixes:
- Restart terminal as normal user or administrator.
- Reinstall node_modules.
- Ensure antivirus/Windows security is not blocking esbuild.
- Move project out of OneDrive if file locking continues.

27. How To Add Features
-----------------------

Add a new backend route:
1. Open backend/apis.py.
2. Add a FastAPI route.
3. Add any request/response Pydantic models.
4. If frontend needs it, add a function in frontend/src/services/api.ts.
5. Use that function from a component or hook.

Add a new database tool for Gemini:
1. Open backend/src/tools.py.
2. Add a method to DatabaseTools.
3. Open backend/src/agent.py.
4. Add the function declaration in _define_tools.
5. Add execution logic in _execute_tool.
6. Mention tool usage in the system prompt if needed.

Change SQL validation rules:
1. Open backend/config.py for blocked keywords/default limit.
2. Open backend/src/tools.py.
3. Edit validate_sql.

Change frontend result display:
1. Open frontend/src/components/ResultTable.tsx.
2. Open frontend/src/components/SQLViewer.tsx if SQL display changes.

Change natural language request behavior:
1. Open frontend/src/hooks/useQuerySimulation.ts.
2. Open backend/src/agent.py.

29. Mental Model Of The Whole App
---------------------------------

Think of the project as four layers:

Layer 1: Frontend UI
- User types a question or SQL.
- User views results.
- User views database schema.

Layer 2: Frontend API client
- Converts UI actions into HTTP calls.
- Lives in frontend/src/services/api.ts.

Layer 3: FastAPI backend
- Receives HTTP requests.
- Owns app startup/shutdown.
- Calls SQLAgent and DatabaseTools.
- Lives mainly in backend/apis.py.

Layer 4: Intelligence and database
- SQLAgent talks to Gemini.
- DatabaseTools talks to Neon PostgreSQL.
- SQLAgent coordinates tool calls.
- DatabaseTools enforces read-only database safety.

In one sentence:
React collects the user's question, FastAPI passes it to SQLAgent, Gemini decides what PostgreSQL to run, DatabaseTools safely executes it on Neon, and React displays the returned SQL and rows.
